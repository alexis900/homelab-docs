# RFC-ID:

RFC-2024-0004-NET

# Título:

Instalación de certificados TLS en los servidores DNS

# Tipo de cambio:

NET

# Fecha propuesta:

2025-07-17

# Estado:

Completada

# Criticidad:

Media

# Justificación de criticidad:

Aunque el cambio no afecta directamente al servicio de resolución DNS,
la interfaz de administración es crítica para la gestión segura del
sistema. La falta de cifrado expone credenciales y configuraciones a
posibles ataques de interceptación en redes comprometidas. Dado que los
servidores se encuentran en una VLAN aislada y el acceso está
restringido, el riesgo está mitigado parcialmente, pero sigue siendo
relevante para la seguridad general del sistema.

# Resumen:

Este RFC describe la instalación de certificados TLS en los servidores
DNS (10.0.30.10 y 10.0.30.11) para asegurar el acceso a la interfaz web
de administración.

# Motivación:

Actualmente, la interfaz web de los servidores DNS no utiliza conexiones
cifradas. Esto representa un riesgo de seguridad al gestionar
remotamente la configuración.

# Dispositivo(s) afectados:

-   10.0.30.10 (dns1.home.arpa)

-   10.0.30.11 (dns2.home.arpa)

# Plan de acción:

-   Acceder a los servidores DNS (dns1.home.arpa y dns2.home.arpa) vía
    SSH.\
    Generar o importar certificados TLS válidos desde la CA interna.

-   Configurar el servidor web para utilizar TLS en la interfaz de
    administración.

-   Verificar el acceso a través de HTTPS desde la VLAN de
    administración.

-   Documentar los cambios aplicados y almacenar copia segura de los
    certificados.

# Riesgos:

-   Posible pérdida de acceso a la interfaz web si la configuración TLS
    es incorrecta.

-   Certificados mal configurados pueden impedir la carga de la
    interfaz.

-   Errores en la CA o en la cadena de confianza podrían generar
    advertencias en los navegadores.

# Mitigación:

-   Realizar una copia de seguridad de la configuración web antes de
    aplicar cambios.

-   Probar los certificados en un entorno de staging o máquina local
    antes de aplicarlos en producción.

-   Mantener acceso SSH habilitado para recuperación manual.

-   Utilizar certificados firmados por una CA interna previamente
    desplegada en los dispositivos de administración.

# Resultado:

Este cambio permitirá una futura implementación de DNS-over-TLS o
DNS-over-HTTPS de forma segura y cifrada, protegiendo así las consultas
internas y asegurando la confidencialidad en la gestión de los
servidores DNS.
