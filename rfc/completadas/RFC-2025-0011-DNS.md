# RFC-ID: {#rfc-id .unnumbered}

RFC-2025-0011-DNS

# Título: {#título .unnumbered}

Migración del servicio DNS local de Pi-hole a Bind9.

# Tipo de **cambio**: {#tipo-de-cambio .unnumbered}

DNS / SRV / INFRA

# Fecha propuesta: {#fecha-propuesta .unnumbered}

2025-08-01

# Estado: {#estado .unnumbered}

Completado

# Criticidad: {#criticidad .unnumbered}

Media

# Justificación de criticidad: {#justificación-de-criticidad .unnumbered}

El cambio implica sustituir el servidor DNS principal de la red
doméstica, afectando a la resolución de nombres y al filtrado de
publicidad. Un error puede provocar interrupciones en servicios internos
y externos.

# Resumen: {#resumen .unnumbered}

Se propone migrar el servicio DNS local desde Pi-hole a Bind9,
manteniendo la funcionalidad de bloqueo de anuncios mediante RPZ o zonas
específicas y gestionando las zonas internas (home.arpa) para la red
doméstica.

# Motivación: {#motivación .unnumbered}

Mejorar la escalabilidad, flexibilidad y control avanzado del servicio
DNS, añadiendo funcionalidades que Pi-hole no soporta nativamente, y
unificar la gestión DNS con Bind9.

# Dispositivo(s) afectados: {#dispositivos-afectados .unnumbered}

-   Contenedor ns1 (servidor maestro)

-   Contenedor ns2 (servidor esclavo)

-   Clientes de la red doméstica (configuración DHCP/DNS)

-   Servidor DHCP (OPNsense o similar)\
    Todas las VLANS

# Plan de acción: {#plan-de-acción .unnumbered}

-   Instalar Bind9 en un contenedor (Apollo).

-   Configurar Bind9 como servidor autoritativo para la zona
    [[home.arpa]{.underline}](http://home.arpa).

-   Configurar reenviadores DNS hacia DNS públicos para dominios
    externos.

-   Desactivar Pi-hole como servidor DNS activo tras confirmación.

-   Documentar configuración y backups.

# Riesgos: {#riesgos .unnumbered}

-   Pérdida temporal de resolución DNS en la red interna si la migración
    falla.

-   Posible incompatibilidad con clientes que usen DNS personalizado.

# Mitigación: {#mitigación .unnumbered}

-   Realizar pruebas en entorno PRE o aislado.

-   Mantener Pi-hole en modo escucha pasiva o DNS secundario durante la
    transición.

-   Preparar rollback rápido a Pi-hole si hay algún fallo.

-   Monitorear logs y tráfico DNS durante la migración.

# **Rollback**:  {#rollback .unnumbered}

# En caso de fallo tras cambiar DNS a Bind9:  {#en-caso-de-fallo-tras-cambiar-dns-a-bind9 .unnumbered}

# Renovar configuración DHCP en clientes o reiniciar conexión.

# Verificar que los clientes usan Pi-hole y que la resolución y filtrado funcionan correctamente.

# Apagar o aislar Bind9 temporalmente para evitar conflictos.

# Analizar los logs y causas para corregir antes de reintentar.

# Resultado: {#resultado .unnumbered}

Se migra satisfactoriamente a Bind9, sin bloqueador de anuncios, para
una posible futura integración de bloqueador de anuncios.
