# RFC-ID:

RFC-2025-0016-NET

# Título:

Creación del servidor DNS secundario NS2

# Tipo de cambio:

NET

# Fecha propuesta:

2025-09-27

# Estado:

Completado

# Criticidad:

Alta

# Justificación de criticidad:

La ausencia de NS2 elimina la redundancia de DNS, aumentando el riesgo
de fallos en la resolución de nombres para servicios internos y
externos. Restaurar NS2 es crítico para la disponibilidad de la
infraestructura.

# Resumen:

Instalación y configuración de un nuevo servidor NS2 que actúe como
secundario de NS1, restaurando la replicación de zonas DNS y la
redundancia de resolución de nombres.

# Motivación:

Garantizar continuidad operativa, restaurar redundancia DNS y reducir
riesgos de interrupción de servicios.

# Dispositivo(s) afectados:

-   Nuevo servidor NS2

-   Clientes que dependen de NS2 como DNS secundario

# Plan de acción:

-   Aprovisionar un servidor nuevo con IP y hostname asignados para NS2.

-   Instalar software DNS (BIND9 u otro estándar usado en la
    infraestructura).

-   Configurar NS2 como secundario de NS1 (type slave; masters { IP_NS1;
    };) en named.conf.

-   Configurar archivos de zona y permisos correctos para replicación de
    NS1.

-   Abrir puertos y permitir tráfico DNS desde NS1 y clientes.

-   Iniciar servicio DNS y validar que las zonas se sincronicen
    correctamente.

-   Realizar pruebas de resolución desde clientes internos y externos.

-   Documentar IP, hostname y configuraciones en inventario de
    infraestructura.

-   Monitorizar NS2 durante 24 horas para asegurar estabilidad y
    replicación correcta.

# Riesgos:

-   Error en la replicación de zonas si se configura incorrectamente.

-   Conflictos de IP/hostname si no se respeta la planificación.

-   Impacto temporal en resolución DNS mientras NS2 no está operativo.

# Mitigación:

-   Validar configuración antes de iniciar el servicio.

-   Mantener NS1 operativo como único DNS hasta que NS2 esté activo.

-   Comunicaciones a equipos afectados sobre la ventana de
    mantenimiento.

-   Backup de NS1 previo a la creación de NS2 como referencia.

# Resultado:

Creación de la CT NS2 en el servidor Apollo con el servicio funcionando
correctamente.
