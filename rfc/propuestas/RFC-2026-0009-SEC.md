# RFC — Caddy en CT por nodo Proxmox para front de servicios internos

## Resumen

Desplegar un CT de Caddy por nodo Proxmox (Zeus y Hermes) como reverse proxy/tls terminator para **servicios internos** (paneles, apps, APIs). Cada CT vive en VLAN 99 (gestión), sirve HTTPS para las zonas internas y gestiona certificados (CA interna/DNS-01). No se almacenarán claves en Git; el storage de Caddy se mantendrá local con opción de sincronización segura (rsync) desde un nodo maestro al secundario.

## Motivación

- Unificar terminación TLS y políticas (HTTP/2/3, headers de seguridad) para todos los servicios internos.
- Reducir dependencia de NPM en DMZ para tráfico de administración.
- Simplificar la renovación de certificados internos sin exponer secretos.

## Alcance / Sistemas afectados

- CT Caddy en Zeus (VLAN 99, IP fija 10.0.99.x) y CT Caddy en Hermes (VLAN 99, IP fija 10.0.99.x).
- Servicios internos publicados: Proxmox (8006), PBS, Omada, UniFi, Uptime Kuma, Home Assistant (interno), NPM (panel), DNS UI si aplica.
- VLAN 99 y clientes de gestión/VPN. Sin exposición WAN.

## Plan de acción

1) Preparar CTs en Proxmox  
   - Crear CT “caddy-zeus” en Zeus y “caddy-hermes” en Hermes (RAM/CPU ligera, disco para `/var/lib/caddy`).  
   - Interfaces en VLAN 99 con IP fija (reservar DHCP o estática).  
   - Cortafuegos Proxmox: permitir 80/443 desde 10.0.99.0/24 en cada CT.

2) DNS interno  
   - Asegurar registros A/CNAME para servicios: `zeus.home.arpa`, `hermes.home.arpa`, `omada.home.arpa`, `unifi.home.arpa`, `uptime.home.arpa`, `hass.home.arpa`, etc.  
   - Añadir `caddy-zeus.home.arpa` y `caddy-hermes.home.arpa` (IPs de los CT) como fronts; opcional `caddy.home.arpa` con failover/RR si se desea.

3) Caddy instalación y configuración  
   - Instalar Caddy binario oficial en cada CT (systemd, `admin 127.0.0.1:2019`).  
   - `Caddyfile` por nodo con vhosts; backends pueden apuntar a servicios locales o comunes.  
   - TLS: `tls internal` (CA interna) o DNS-01 con zona interna; `strict_sni_host`; bind solo en 10.0.99.0/24.  
   - Opcional: mTLS o basicauth para paneles críticos.

4) Storage y sincronización  
   - Mantener `/var/lib/caddy` permisos 700/600.  
   - Designar Zeus como maestro de certificados y sincronizar a Hermes vía rsync pull con usuario dedicado y clave restringida; recarga de Caddy tras sync.

5) Integración y monitorización  
   - Añadir checks HTTPS en Uptime Kuma para cada vhost (en ambos fronts si aplican).  
   - Logs a fichero y, si procede, envío a Loki/promtail.

6) Documentación  
   - Actualizar `Doc-Red/Servicios.md` con la capa Caddy y sus CTs.  
   - Registrar reglas de firewall en `Doc-Red/Seguridad.md`.  
   - Añadir entradas DNS nuevas en el inventario.

## Riesgos

- Exposición indebida si se abren reglas fuera de VLAN 99.  
- Desincronización de certificados si se usa HA sin coordinar storage.  
- Fallo de Caddy podría afectar varios servicios a la vez.

## Mitigación

- Bind solo a 10.0.99.x y clientes VPN; mTLS/basicauth en paneles.  
- Nodo secundario solo consume storage sincronizado; renovaciones centralizadas en el maestro.  
- Mantener acceso directo por IP/puerto a servicios críticos como fallback durante despliegue.

## Plan de rollback

1. Deshabilitar Caddy y revertir DNS hacia endpoints directos (puertos nativos).  
2. Restaurar reglas de firewall previas.  
3. Eliminar o aislar el CT de Caddy si causa impacto.

## Validaciones post-cambio

- [ ] Acceso HTTPS a cada vhost con certificado válido.  
- [ ] Paneles críticos accesibles y autenticación reforzada (mTLS/basicauth si aplica).  
- [ ] Uptime Kuma sin alertas.  
- [ ] Acceso directo por IP/puerto sigue disponible desde VLAN 99 como respaldo.

## Cambios DNS (si aplica)

- Entrada: `caddy.home.arpa` A 10.0.99.x (front central).  
- Validación: `dig caddy.home.arpa @ns1` / acceso HTTPS ok.  
- Inventario actualizado en `Doc-Red/Servicios.md`.
