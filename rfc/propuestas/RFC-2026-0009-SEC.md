# RFC — Despliegue de Caddy por nodo Proxmox para proxy de gestión en VLAN 99

## Resumen

Instalar y configurar Caddy en un CT dedicado por nodo Proxmox (Zeus, Hermes) para exponer servicios de gestión internos vía HTTPS con certificados automatizados (CA interna/DNS-01), limitado a VLAN 99 y con opción de mTLS/basicauth. Se evitará usar repos Git para claves; el almacenamiento de certificados se gestionará localmente y se podrá sincronizar entre nodos con rsync seguro.

## Motivación

- Unificar terminación TLS y headers modernos para paneles (Proxmox, PBS, servicios auxiliares).
- Reducir dependencia del NPM en DMZ para accesos de administración.
- Simplificar renovación de certificados internos sin exponer secretos.

## Alcance / Sistemas afectados

- CTs de Caddy en cada nodo Proxmox: Zeus (CT en 10.0.99.x), Hermes (CT en 10.0.99.x).
- Servicios gestionados localmente en cada nodo (Proxmox 8006, PBS si aplica).
- VLAN 99 únicamente. Sin exposición WAN.

## Plan de acción

1. Preparar DNS internos: `zeus.home.arpa`, `hermes.home.arpa` ya existentes; añadir otros hostnames locales si aplican (p.ej. `pbs.home.arpa`).
2. Desplegar CT “caddy” en cada nodo (VLAN 99, IP fija 10.0.99.x, sin acceso WAN entrante).
3. Instalar Caddy binario oficial en cada CT (systemd, admin API en 127.0.0.1:2019).
4. Configurar `Caddyfile` por nodo:
   - vhosts para paneles locales (`https://127.0.0.1:8006` etc.).
   - `tls internal` o DNS-01 con CA interna; restringir bind a `10.0.99.0/24`.
   - Opcional mTLS/basicauth en rutas sensibles.
5. Storage y backups:
   - Mantener `/var/lib/caddy` con permisos 700/600.
   - Opcional: rsync pull desde Zeus a Hermes con usuario dedicado y clave restringida; sin Git.
6. Uptime Kuma: añadir checks HTTPS a los vhosts de gestión.
7. Documentar en `Doc-Red/Servicios.md` (sección Caddy) y reglas en `Doc-Red/Seguridad.md` si se abren puertos específicos.

## Riesgos

- Exposición de paneles si se relajan reglas o binds; riesgo mitigable con mTLS/ACL.
- Desincronización de certificados si cada nodo renueva por su cuenta y se comparten nombres.
- Error de configuración podría interrumpir acceso 8006 (fallback necesario).

## Mitigación

- Limitar escucha a VLAN 99 y VPN; aplicar mTLS/basicauth.
- Designar un nodo maestro de certificados si se sincroniza storage; scripts con `--delete` controlado.
- Mantener acceso directo a 8006 por IP como backup durante despliegue.

## Plan de rollback

1. Detener servicio Caddy y deshabilitarlo.
2. Restaurar acceso directo a puertos nativos (8006/8007) desde VLAN 99.
3. Revertir cambios DNS si se añadieron hostnames nuevos.

## Validaciones post-cambio

- [ ] Acceso HTTPS a `https://zeus.home.arpa` y `https://hermes.home.arpa` con certificado válido.
- [ ] mTLS/basicauth funcionando en rutas protegidas (si aplica).
- [ ] Acceso directo a 8006 sigue disponible desde VLAN 99.
- [ ] Uptime Kuma en verde para vhosts de gestión.

## Cambios DNS (si aplica)

- Entrada: `<host>.home.arpa` A `10.0.99.x` (motivo: front Caddy nodo <host>).
- Validación: `dig <host>.home.arpa @ns1` / acceso HTTPS ok.
- Inventario actualizado en `Doc-Red/Servicios.md`.

- Evaluar storage compartido (MinIO/S3) si más nodos requieren los mismos certs.
- Considerar mTLS obligatorio para paneles críticos.
