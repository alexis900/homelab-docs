# RFC — Instalación de Authentik (SSO/IdP) para servicios internos

## Resumen

Se propone desplegar **Authentik** como proveedor de identidad (SSO/IdP) para centralizar autenticación y aplicar políticas de acceso (MFA/2FA, grupos y flujos) a servicios internos.

## Motivación

- Centralizar credenciales y reducir contraseñas locales en cada servicio.
- Habilitar 2FA y políticas de acceso consistentes.
- Mejorar trazabilidad de accesos administrativos.

## Alcance / Sistemas afectados

- Servicio Authentik (UI + workers + base de datos) en **VLAN 30 (servicios internos)**.
- **LDAP Outpost** de Authentik para servicios que requieran LDAP.
- Proxy inverso **Caddy** para publicación interna segura (ver RFC de Caddy existente).
- DNS interno (NS1/NS2) para nuevo hostname.
- Integraciones OIDC/SAML en servicios que lo soporten (por fases).

Excluido: publicación externa a Internet (solo interno salvo aprobación explícita).

## Plan de acción

1. Preparar CT/host para Authentik (Docker/Compose) en **VLAN 30**.
2. Desplegar Authentik con Postgres y Redis según documentación oficial.
3. Configurar reverse proxy en **Caddy** con HTTPS interno.
4. Crear DNS interno `auth.home.arpa` apuntando al servicio.
5. Configurar 2FA y política base (grupos, usuarios admin).
6. Integrar por fases servicios internos (OIDC/SAML): Proxmox, Uptime Kuma, NPM, etc. (según compatibilidad).
7. Monitorización básica en Uptime Kuma.
8. Habilitar **LDAP Outpost** si algún servicio requiere LDAP.

## Riesgos

- Pérdida temporal de acceso si se fuerza SSO sin fallback local.
- Configuración incorrecta de OIDC/SAML que bloquee el login.
- Dependencias nuevas (Postgres/Redis) aumentando superficie operativa.

## Mitigación

- Mantener cuentas locales de emergencia durante la migración.
- Implementar SSO por servicio con ventana de rollback.
- Backups de DB y snapshots del CT antes de cambios críticos.

## Plan de rollback

1. Deshabilitar SSO en el servicio afectado y restaurar auth local.
2. Revertir cambios de NPM/DNS si es necesario.
3. Restaurar snapshot del CT o backups si el despliegue falla.

## Validaciones post-cambio

- [ ] UI de Authentik accesible por HTTPS interno.
- [ ] Login admin funcional con 2FA.
- [ ] Healthchecks sin errores en logs.
- [ ] Integración OIDC/SAML validada en al menos un servicio piloto.
- [ ] Monitorización en verde (Uptime Kuma).

## Cambios DNS (si aplica)

- Entrada: `auth.home.arpa` `A` `<IP pendiente>` (motivo: acceso interno a Authentik)
- Validación: `dig auth.home.arpa @ns1` / UI accesible
- Inventario actualizado en `Doc-Red/Servicios.md`
